## はじめに

習慣トラッカーを自作したことがある人なら、一度はこの問題にぶつかるはずだ。ストリークが途切れた瞬間、ユーザーがアプリを開かなくなる。通知を送れば送るほど開封率が下がる。「継続を支援するはずのツールが、継続を壊している」という状況だ。

この記事は、そのメカニズムをDB設計と通知設計の両面から分解して、実際に自分が使っている構造を共有する。完璧な答えではないが、少なくともストリーク切れでユーザーが消える問題は、設計レベルで緩和できる。

---

## 結論先出し

- ストリークは「連続日数」ではなく「回復可能な状態機械」として持つ
- 通知は「やれ」ではなく「状態の観察結果」として送る
- DBには「最後に記録した日」と「猶予フラグ」を分けて持つ

この3点を実装するだけで、ストリーク切れによる離脱率はかなり変わる。自分がALTERを作るときに実際に試した構造なので、実感を持って書ける。

---

## 問題の深掘り：なぜストリーク切れでユーザーは消えるのか

Habiticaを使っていたことがある。3ヶ月続いたストリークが、出張中に1日記録を忘れただけでゼロに戻った。その瞬間、アプリを開く理由が消えた感覚があった。

これは意志の問題ではなく、設計の問題だと今は思っている。

習慣トラッカーの多くは、ストリークを「連続日数のカウンタ」として実装する。シンプルで直感的だが、このモデルには致命的な欠陥がある。**失敗のコストが非線形に大きい**のだ。

1日サボると、積み上げた全記録の「意味」が消える。人間の損失回避バイアスとこれが組み合わさると、「どうせまたゼロになるなら記録しなくていい」という判断が生まれる。行動経済学の話をしたいわけじゃないが、実際にユーザーがそう動くのは数字でも見えている。

自分がALTERの初期版を作ったとき、ストリークを単純カウンタで実装した。90日チャレンジのドッグフーディング中に、自分自身が3日記録を飛ばして「もういいか」と思った。作った本人がそう感じたので、これは設計の問題だと確信した。

もう一つの問題が通知だ。「今日の記録をしてください」という通知は、最初の数日は効く。1週間後には無視される。2週間後にはミュートされる。これも意志の問題ではなく、通知の設計が「命令」になっているからだ。

命令型の通知は、ユーザーを「できていない自分」に直面させる。その感覚が不快なので、通知ごとアプリを避けるようになる。通知を送るほど開封率が下がる、という逆説が生まれる。

この2つの問題、ストリーク設計と通知設計は、DBの構造から変えないと根本的には解決しない。

## 解決手順

### ステップ1：ストリークを「状態機械」として設計する

まず、ストリークを「連続日数」ではなく「状態」として持つことを考える。

状態は以下の4つで十分だ。

| 状態 | 意味 |
|------|------|
| `active` | 直近N日以内に記録あり |
| `grace` | 猶予期間中（記録なしだが、まだリセットしていない） |
| `broken` | 猶予期間を超えてリセット済み |
| `recovering` | broken後に記録を再開した |

`grace`の存在がポイントだ。1日記録を忘れても、即座にストリークをゼロにしない。猶予期間（自分の場合は24〜48時間）を設けて、その間に記録が来たら`active`に戻す。

DBのテーブル設計はこうなる。

```sql
CREATE TABLE streak_states (
    id          BIGINT PRIMARY KEY,
    user_id     BIGINT NOT NULL,
    habit_id    BIGINT NOT NULL,
    state       VARCHAR(16) NOT NULL DEFAULT 'active',
    streak_count INT NOT NULL DEFAULT 0,
    last_logged_at  TIMESTAMP,
    grace_expires_at TIMESTAMP,
    updated_at  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

`last_logged_at`と`grace_expires_at`を分けて持つのが重要だ。「最後に記録した時刻」と「猶予が切れる時刻」は別の概念で、混在させると状態遷移のロジックが壊れる。

状態遷移の処理はこうなる（擬似コードだが、実際にこの構造で動かしている）。

```c
typedef enum {
    STREAK_ACTIVE,
    STREAK_GRACE,
    STREAK_BROKEN,
    STREAK_RECOVERING
} StreakState;

typedef struct {
    int64_t  user_id;
    int64_t  habit_id;
    StreakState state;
    int      streak_count;
    time_t   last_logged_at;
    time_t   grace_expires_at;
} StreakRecord;

StreakState transition_streak(StreakRecord *rec, time_t now, int logged_today)
{
    if (logged_today) {
        if (rec->state == STREAK_BROKEN || rec->state == STREAK_RECOVERING) {
            rec->state = STREAK_RECOVERING;
            rec->streak_count = 1;
        } else {
            rec->state = STREAK_ACTIVE;
            rec->streak_count += 1;
        }
        rec->last_logged_at = now;
        rec->grace_expires_at = now + 48 * 3600; /* 48時間猶予 */
        return rec->state;
    }

    /* 記録なしの場合 */
    if (now < rec->grace_expires_at) {
        rec->state = STREAK_GRACE;
        return rec->state;
    }

    /* 猶予切れ */
    rec->state = STREAK_BROKEN;
    rec->streak_count = 0;
    return rec->state;
}
```

C言語で書いたのは、この設計がバックエンドのバッチ処理（状態更新ジョブ）として動くことを想定しているからだ。実際のALTERはNode.js＋PostgreSQLで動いているが、状態遷移のロジックはこの構造をそのまま移植している。

### ステップ2：通知を「命令」から「観察」に変える

通知の文面を変える前に、「いつ送るか」を変える必要がある。

命令型の通知は「記録していない → 送る」というトリガーで動く。これが問題だ。

観察型の通知は「状態が変化した → 送る」というトリガーで動く。具体的にはこうなる。

| 状態変化 | 通知内容 |
|----------|----------|
| `active` → `grace` | 「昨日の記録が来ていない。今日中に入れると続きになる」 |
| `grace` → `active` | 「記録を確認した。ストリーク継続中」 |
| `grace` → `broken` | 「ストリークがリセットされた。次の記録から再スタート」 |
| `broken` → `recovering` | 「記録が来た。再開した」 |
| `recovering` → `active`（7日継続） | 「7日続いた」 |

文面の違いに気づくかもしれない。「してください」という命令がない。状態を観察して、事実を伝えるだけだ。

これは意図的な設計で、「やれ」という通知はユーザーを「できていない自分」に直面させるが、「こういう状態になっている」という通知は情報提供に過ぎない。受け取り手の心理的負荷が違う。

通知のスケジューリングはDBのジョブキューで管理する。

```sql
CREATE TABLE notification_jobs (
    id              BIGINT PRIMARY KEY,
    user_id         BIGINT NOT NULL,
    habit_id        BIGINT NOT NULL,
    trigger_state   VARCHAR(16) NOT NULL,
    scheduled_at    TIMESTAMP NOT NULL,
    sent_at         TIMESTAMP,
    payload         JSONB
);
```

`trigger_state`に「どの状態変化で発火したか」を持たせることで、同じ状態変化に対して重複通知が送られるのを防げる。`sent_at`がNULLのレコードをバッチで処理するだけでいい。

### ステップ3：「ストリーク数」の表示を変える

DBの設計が変わったら、UIの表示も変える必要がある。

`streak_count`をそのまま表示するのをやめた。代わりに「過去30日の記録率」を表示する。

```sql
SELECT
    COUNT(*) FILTER (WHERE logged_date >= NOW() - INTERVAL '30 days') AS logged_days,
    30 AS total_days,
    ROUND(
        COUNT(*) FILTER (WHERE logged_date >= NOW() - INTERVAL '30 days') * 100.0 / 30,
        1
    ) AS rate_30d
FROM habit_logs
WHERE user_id = $1 AND habit_id = $2;
```

「30日中23日記録した（76.7%）」という表示は、「ストリーク0日」という表示より遥かに正確で、遥かに継続意欲を削がない。1日サボっても76.7%が75.0%になるだけだ。

## 落とし穴と対処

### 落とし穴1：猶予期間をユーザーが知らないと逆効果になる

`grace`状態の存在をユーザーに見せないと、「通知が来たのに記録しなかったらどうなるの？」という不安が残る。これは隠すより見せた方がいい。

「あと18時間以内に記録するとストリーク継続」という表示を入れたら、grace期間中の記録率が上がった。数字は出せないが、体感として明らかに変わった。

### 落とし穴2：`grace_expires_at`のタイムゾーン処理

これは実際にやらかした。`grace_expires_at`をUTCで保存していたが、ユーザーのタイムゾーンを考慮せずに「今日中」という表示をしていた。日本時間で23時に記録したユーザーが、UTCでは翌日扱いになってストリークが切れる、という問題が出た。

対処は単純で、ユーザーのタイムゾーンをプロフィールに持たせて、`grace_expires_at`の計算時に使うだけだ。ただし「ユーザーの1日の終わり」をどこに設定するかは、アプリの性質によって変わる。自分は「ユーザーのローカル時間で翌日の午前3時」を猶予期限にしている。深夜に記録する人を拾うためだ。

### 落とし穴3：`recovering`状態の扱いを決めておかないとUIが壊れる

`broken`後に`recovering`になったとき、ストリーク数を「1」から始めるか、「0」から始めるかを最初に決めておかないと、表示ロジックが複数箇所で矛盾する。自分は「記録した日を1日目」として`streak_count = 1`から始めることにした。「再スタートした」という感覚をポジティブに演出するためだ。

## まとめ

ストリーク設計の問題は、「連続日数カウンタ」というシンプルすぎるモデルにある。状態機械として設計し直すだけで、1日の記録忘れがユーザー離脱に直結する構造を壊せる。通知は「命令」から「観察結果の報告」に変えると、開封率の低下が緩和される。どちらも実装コストは低いが、DBの設計段階で決めておかないと後から直すのが面倒になる。最初から状態を分けて持つことを勧める。
