## この記事で何ができるようになるか

C言語は書けるのに、「このコードは安全か？」と聞かれると答えに詰まる——そういう人は多い。

この記事では、C言語の安全性を体系的に評価するための**4層モデル（L0〜L3）**を紹介する。読み終えるころには「どの層で何を確認すればいいか」が分かり、自分のコードや他人のコードを見るときの視点が変わる。

難しい話は後回しにして、まず「動かして確認する」ことを優先する。

---

## 前提知識

### 必要なもの
- **C言語の基礎**（変数・ポインタ・配列・関数が読める程度）
- **gccが動く環境**（Linux/macOS/WSL のいずれか。インストール済みであること）

### 必要ないもの
- セキュリティの専門知識：この記事では使わないので心配無用
- MISRA-CやCERTなどの規格の知識：名前だけ知っていれば十分
- 大規模プロジェクトの経験：数十行のサンプルコードで練習できる

### あると理解が早いもの
- Valgrindの使用経験：なくてもOKだが、あれば「L2の章」が30分早く理解できる

## 4層モデルとは何か：全体像を先に把握する

4層モデルは、C言語コードの安全性を**粒度別に分けて評価する**ための枠組みだ。

| 層 | 名称 | 一言で言うと |
|---|---|---|
| **L0** | 構文・コンパイル層 | コンパイラが検出できるレベル |
| **L1** | 実行時安全層 | 動かしてみて初めて分かるレベル |
| **L2** | リソース管理層 | メモリ・ファイル・ロックの管理 |
| **L3** | 設計・仕様整合層 | コードが仕様通りか、意図通りか |

**下の層（L0）ほど機械的に検出しやすく、上の層（L3）ほど人間の判断が必要になる。**

最初は「L0が全部クリアできている」状態を目標にする。そこから順番に上げていく。

```
L3 ─── 設計・仕様整合（人間の判断が必要）
 │
L2 ─── リソース管理（ツール＋人間）
 │
L1 ─── 実行時安全（ツールで自動化できる）
 │
L0 ─── 構文・コンパイル（コンパイラ＋静的解析）
```

## ステップ1：L0 ── コンパイラに怒らせる

### やること
コンパイラの警告を最大限に有効化して、L0レベルの問題を洗い出す。

### 手順

**1. 以下のサンプルコードを用意する**

```c
/* sample_l0.c */
#include <stdio.h>

int add(int a, int b) {
    return a + b;
}

int main(void) {
    int result;          /* 未初期化変数 */
    int x = 10;
    
    if (x = 5) {         /* 代入を条件式に使っている（意図的なバグ） */
        result = add(x, 20);
    }
    
    printf("result: %d\n", result);
    return 0;
}
```

**2. まず「普通に」コンパイルしてみる**

```bash
gcc sample_l0.c -o sample_l0
./sample_l0
```

おそらく動く。エラーも出ない。でもこのコードには問題が2つある。

**3. 警告を最大化してコンパイルし直す**

```bash
gcc -Wall -Wextra -Wpedantic sample_l0.c -o sample_l0
```

以下のような警告が出るはずだ：

```
sample_l0.c:12:9: warning: suggest parentheses around assignment
    used as truth value [-Wparentheses]
     if (x = 5) {
         ^
sample_l0.c:7:9: warning: 'result' may be used uninitialized
    [-Wmaybe-uninitialized]
```

**4. 警告の意味を読む**

- `suggest parentheses around assignment`：`if (x = 5)` は「代入」であって「比較」ではない。意図的なら `if ((x = 5))` と書くべき。おそらくバグ。
- `'result' may be used uninitialized`：`result` が初期化されないまま使われる可能性がある。

**5. 修正する**

```c
/* 修正版 */
int main(void) {
    int result = 0;      /* 初期化 */
    int x = 10;
    
    if (x == 5) {        /* 比較演算子に修正 */
        result = add(x, 20);
    }
    
    printf("result: %d\n", result);
    return 0;
}
```

### 成功の目安
`-Wall -Wextra` で警告ゼロになれば、L0クリア。

### もしうまくいかない場合
- **gccが入っていない**：`which gcc` で確認。Ubuntuなら `sudo apt install build-essential`
- **警告が英語で読めない**：Google翻訳でそのまま貼り付けてOK。慣れれば読めるようになる

## ステップ2：L1 ── 実行時の境界を確認する

### やること
コンパイルは通るが実行時に壊れるパターンを、ツールを使って検出する。

### 手順

**1. 境界外アクセスのサンプルを用意する**

```c
/* sample_l1.c */
#include <stdio.h>
#include <string.h>

void copy_name(char *dst, const char *src) {
    strcpy(dst, src);   /* 長さチェックなし */
}

int main(void) {
    char buf[8];
    copy_name(buf, "Hello, World!");  /* 8バイトに13文字は入らない */
    printf("%s\n", buf);
    return 0;
}
```

**2. AddressSanitizerで実行する**

```bash
gcc -fsanitize=address -g sample_l1.c -o sample_l1
./sample_l1
```

出力例：

```
==1234==ERROR: AddressSanitizer: stack-buffer-overflow on address 0x...
WRITE of size 14 at 0x... thread T0
    #0 0x... in copy_name sample_l1.c:6
    #1 0x... in main sample_l1.c:11
```

**3. 修正する**

```c
#include <stdio.h>
#include <string.h>

void copy_name(char *dst, size_t dst_size, const char *src) {
    strncpy(dst, src, dst_size - 1);
    dst[dst_size - 1] = '\0';   /* 終端保証 */
}

int main(void) {
    char buf[8];
    copy_name(buf, sizeof(buf), "Hello, World!");
    printf("%s\n", buf);
    return 0;
}
```

**4. 再実行して確認**

```bash
gcc -fsanitize=address -g sample_l1.c -o sample_l1
./sample_l1
# => Hello, （オーバーフローなし）
```

### 成功の目安
AddressSanitizerのエラーが出なくなれば、L1クリア。

### もしうまくいかない場合
- **`-fsanitize=address` が使えない**：gccのバージョンが古い可能性。`gcc --version` で確認。4.8以降なら使える
- **macOSでエラーが違う形式になる**：clangでも同じオプションが使える

## ステップ3：L2 ── リソースの生死を追う

### やること
メモリやファイルが「確保したら必ず解放されているか」を確認する。

### 手順

**1. メモリリークのサンプルを用意する**

```c
/* sample_l2.c */
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

char *create_message(const char *name) {
    char *msg = malloc(64);
    if (msg == NULL) return NULL;
    snprintf(msg, 64, "Hello, %s!", name);
    return msg;
}

int main(void) {
    char *m1 = create_message("Alice");
    char *m2 = create_message("Bob");
    
    printf("%s\n", m1);
    /* m1 を free し忘れている */
    
    printf("%s\n", m2);
    free(m2);
    
    return 0;
}
```

**2. Valgrindで確認する**

```bash
gcc -g sample_l2.c -o sample_l2
valgrind --leak-check=full ./sample_l2
```

出力例：

```
LEAK SUMMARY:
   definitely lost: 64 bytes in 1 blocks
```

**3. L2チェックリストで確認する（手動）**

ツールだけでなく、以下を目で確認する習慣をつける：

```
□ malloc/calloc/realloc の後に free があるか
□ fopen の後に fclose があるか
□ エラー時の早期 return でリソースが解放されているか
□ 関数をまたいだ所有権が明確か（誰が free するか）
```

**4. 修正する**

```c
int main(void) {
    char *m1 = create_message("Alice");
    char *m2 = create_message("Bob");
    
    if (m1) {
        printf("%s\n", m1);
        free(m1);   /* 追加 */
    }
    
    if (m2) {
        printf("%s\n", m2);
        free(m2);
    }
    
    return 0;
}
```

### 成功の目安
`valgrind` の `definitely lost: 0 bytes` になれば、L2クリア。

### もしうまくいかない場合
- **Valgrindが入っていない**：`sudo apt install valgrind`（macOSでは代替としてAddressSanitizerの `--leak-check` を使う）
- **`possibly lost` が残る**：最初は `definitely lost` だけを0にすることを目標にする

## ステップ4：L3 ── 「仕様通りか」を問う

### やること
L3は機械的に検出できない。「このコードは何をすべきか」と「実際に何をしているか」を照合する。

### 手順

**L3チェックの考え方**

L3では以下の問いに答える：

```
1. この関数の事前条件は何か？（NULLを渡してはいけない、など）
2. この関数の事後条件は何か？（戻り値が0なら成功、など）
3. エラー時の動作は仕様に書かれているか？
4. 並行実行された場合の動作は保証されているか？
```

**L3チェックのコード例**

```c
/**
 * @brief バッファにデータをコピーする
 * @pre   dst != NULL, src != NULL, dst_size > 0
 * @post  戻り値が0なら dst は null 終端されている
 * @post  戻り値が-1なら dst の内容は不定
 */
int safe_copy(char *dst, size_t dst_size, const char *src) {
    if (dst == NULL || src == NULL || dst_size == 0) {
        return -1;   /* 事前条件違反 */
    }
    strncpy(dst, src, dst_size - 1);
    dst[dst_size - 1] = '\0';
    return 0;
}
```

コメントに `@pre` と `@post` を書く習慣をつけるだけで、L3の意識が育つ。

### 成功の目安
全関数に「何を受け取り、何を返すか、エラー時はどうなるか」が明文化されていれば、L3の入り口に立てている。

## よくあるエラーとその対処

### エラー1：`-Wall` をつけたら大量の警告が出て途方に暮れる

**原因**：既存コードに長年放置された問題が一気に可視化された。

**対処**：全部を一度に直そうとしない。`-Werror`（警告をエラーに昇格）は最初はつけない。まず「新しく書くコードだけ警告ゼロ」を目標にする。

---

### エラー2：AddressSanitizerが「エラーなし」なのに動作がおかしい

**原因**：AddressSanitizerはメモリの境界違反は検出するが、ロジックのバグは検出しない。

**対処**：L1ツールは「メモリ安全性」の確認だけ。ロジックのバグはL3（仕様との照合）で対処する。

---

### エラー3：`free` を2回呼んでしまった（double free）

**原因**：ポインタを `free` した後に `NULL` を代入していない。

**対処**：
```c
free(ptr);
ptr = NULL;   /* これを習慣にする */
```
AddressSanitizerで `double-free` エラーとして検出される。

---

### エラー4：Valgrindが遅すぎて使えない

**原因**：Valgrindは実行速度が10〜50倍遅くなる。大きなプログラムでは現実的でない。

**対処**：テスト用の小さなユニットテストにだけValgrindを走らせる。CIに組み込むのは`-fsanitize=address`の方が現実的。

---

### エラー5：L0〜L3のどれから始めればいいか分からない

**対処**：必ずL0から始める。L0が通っていないコードにL3の議論をしても意味がない。層は下から順番にクリアする。

## 次に学ぶべきこと

この記事を読んで動かせたら、次のステップ：

1. **静的解析ツールの導入**：`cppcheck` や `clang-tidy` を使って、L0/L1の自動チェックをCIに組み込む
   推奨リソース：`cppcheck` 公式ドキュメント（無料）、[Amazonアソシエイト：「セキュアコーディング in C and C++」](https://www.amazon.co.jp/)

2. **CERT Cコーディング標準の基礎**：L1〜L2の判断基準を体系化したい場合の参照先
   推奨リソース：CERT C Coding Standard（オンライン公開、無料）

3. **監査レポートの書き方**：「どの層で何が問題か」を他人に伝えるドキュメントスキル
   推奨リソース：後述のCSAFフレームワーク

1ヶ月かけて3ステップ進めば、C言語安全性監査の基礎は固まる。最初の3日は「どこを見ればいいか分からない」と感じるかもしれない。1週間で視点が変わってくる。
